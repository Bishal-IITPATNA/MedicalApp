Application Factory Pattern\n- `app/__init__.py` creates Flask app instance\n- Configuration loaded from environment variables\n- Database and JWT initialization\n- Blueprint registration for modular routes\n\n#### 2. Database Layer (SQLAlchemy)\n- **Models**: Define database schema and relationships\n- **Migrations**: Version control for database schema changes\n- **Relationships**: Foreign keys and joins between entities\n\n#### 3. API Layer (Flask-RESTful)\n- **Routes**: Organized by user roles (patient, doctor, admin, etc.)\n- **Authentication**: JWT-based with role-based access control\n- **Validation**: Input validation and error handling\n- **Serialization**: JSON response formatting\n\n#### 4. Business Logic\n- **Services**: Business logic separated from route handlers\n- **Utilities**: Common functions and helpers\n- **Validators**: Input validation and sanitization\n\n### Frontend Architecture (Flutter)\n\n#### 1. Presentation Layer\n- **Screens**: Full-page UI components\n- **Widgets**: Reusable UI components\n- **State Management**: Provider pattern for state management\n\n#### 2. Business Logic Layer\n- **Services**: API communication and business logic\n- **Models**: Data structures and serialization\n- **Providers**: State management providers\n\n#### 3. Data Layer\n- **API Service**: HTTP client wrapper with error handling\n- **Local Storage**: SharedPreferences for token storage\n- **Caching**: In-memory caching for performance\n\n## Database Design\n\n### Core Entities\n\n#### Users Table\n```sql\nusers (\n    id INTEGER PRIMARY KEY,\n    email VARCHAR(120) UNIQUE NOT NULL,\n    password_hash VARCHAR(255) NOT NULL,\n    role VARCHAR(20) NOT NULL,  -- patient, doctor, medical_store, lab_store, admin\n    is_active BOOLEAN DEFAULT TRUE,\n    created_at DATETIME,\n    updated_at DATETIME\n)\n```\n\n#### Patient Profile\n```sql\npatients (\n    id INTEGER PRIMARY KEY,\n    user_id INTEGER REFERENCES users(id),\n    name VARCHAR(100) NOT NULL,\n    phone VARCHAR(15),\n    date_of_birth DATE,\n    gender VARCHAR(10),\n    address TEXT,\n    city VARCHAR(50),\n    state VARCHAR(50),\n    pincode VARCHAR(10),\n    emergency_contact VARCHAR(15),\n    blood_group VARCHAR(5),\n    allergies TEXT,\n    medical_history TEXT\n)\n```\n\n#### Doctor Profile\n```sql\ndoctors (\n    id INTEGER PRIMARY KEY,\n    user_id INTEGER REFERENCES users(id),\n    name VARCHAR(100) NOT NULL,\n    phone VARCHAR(15),\n    specialization VARCHAR(100),\n    qualification VARCHAR(200),\n    experience_years INTEGER,\n    consultation_fee DECIMAL(10,2),\n    available_days TEXT,  -- JSON array\n    available_time_start TIME,\n    available_time_end TIME\n)\n```\n\n### Relationship Design\n\n1. **One-to-One**: User -> Patient/Doctor/Store profiles\n2. **One-to-Many**: Doctor -> Appointments, Patient -> Orders\n3. **Many-to-Many**: Lab Orders -> Lab Tests (through junction table)\n\n### Data Integrity\n\n- Foreign key constraints ensure referential integrity\n- Unique constraints prevent duplicate emails/phones\n- Check constraints validate enum values\n- Triggers for automatic timestamp updates\n\n## API Design Principles\n\n### RESTful Design\n- Resource-based URLs (`/api/patients/123/appointments`)\n- HTTP methods for operations (GET, POST, PUT, DELETE)\n- Status codes for response indication\n- JSON for data exchange\n\n### Authentication Flow\n1. **Login**: POST `/api/auth/login` -> JWT tokens\n2. **Request**: Include `Authorization: Bearer <token>` header\n3. **Validation**: Server validates JWT and extracts user info\n4. **Authorization**: Check user role for endpoint access\n5. **Refresh**: Use refresh token to get new access token\n\n### Error Handling\n- Consistent error response format\n- Appropriate HTTP status codes\n- Detailed error messages for development\n- Generic messages for production\n\n## Security Architecture\n\n### Authentication & Authorization\n- **JWT Tokens**: Short-lived access tokens (24h)\n- **Refresh Tokens**: Long-lived for token renewal (30 days)\n- **Role-Based Access**: Endpoints restricted by user roles\n- **Multi-Role Support**: Users can have multiple roles\n\n### Data Protection\n- **Password Hashing**: PBKDF2 with salt\n- **Input Validation**: Server-side validation for all inputs\n- **SQL Injection Protection**: SQLAlchemy ORM usage\n- **XSS Protection**: Output encoding and CSP headers\n\n### Network Security\n- **HTTPS Only**: TLS encryption in production\n- **CORS Configuration**: Proper cross-origin settings\n- **Rate Limiting**: API endpoint rate limiting\n- **File Upload Security**: Type and size validation\n\n## State Management (Frontend)\n\n### Provider Pattern\n```dart\n// Auth Provider\nclass AuthProvider with ChangeNotifier {\n  User? _user;\n  String? _token;\n  \n  bool get isAuthenticated => _token != null;\n  User? get user => _user;\n  \n  Future<void> login(String email, String password) async {\n    // Login logic\n    notifyListeners();\n  }\n}\n\n// Usage in Widget\nConsumer<AuthProvider>(\n  builder: (context, auth, child) {\n    return auth.isAuthenticated \n      ? Management**: In-memory caching with expiration\n\n## Performance Considerations\n\n### Backend Optimization\n- **Database Indexing**: Proper indexes on frequently queried columns\n- **Query Optimization**: Efficient SQL queries with joins\n- **Connection Pooling**: Database connection reuse\n- **Caching**: Redis for frequently accessed data\n\n### Frontend Optimization\n- **Lazy
